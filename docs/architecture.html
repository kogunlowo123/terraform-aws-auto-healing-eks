Architecture Reference -- Self-Healing EKS Cluster with Automated Node Remediation

Terraform AWS EKS Karpenter Cluster Autoscaler Lambda EventBridge Helm

Executive Summary

This module deploys a production-ready Amazon EKS cluster with comprehensive self-healing capabilities. It provisions managed node groups with launch templates, installs Karpenter for intelligent node auto-provisioning, deploys the Cluster Autoscaler for ASG-based scaling, and sets up an automated node remediation pipeline using EventBridge, Lambda, and CloudWatch Alarms. The Node Termination Handler ensures graceful pod drainage during spot interruptions and scheduled maintenance events. All components integrate with OIDC-based IRSA for secure, least-privilege access.

Component Breakdown

EKS Control Plane

  • EKS Cluster -- Kubernetes 1.27+ with full audit logging (api, audit, authenticator, controllerManager, scheduler)
  • OIDC Provider -- Enables IAM Roles for Service Accounts (IRSA)
  • Security Groups -- Separate cluster and node SGs with minimal rules

EKS Addons

  • CoreDNS -- Cluster DNS resolution
  • kube-proxy -- Network proxy on each node
  • VPC CNI -- Pod networking with VPC IPs
  • EBS CSI Driver -- Persistent volume support

Managed Node Groups

  • Configurable instance types, scaling (min/max/desired), capacity type (ON_DEMAND/SPOT)
  • Launch templates with gp3 encrypted EBS, IMDSv2 required, detailed monitoring
  • Labels and taints for workload scheduling

Auto-Scaling Layer

  • Karpenter v0.34.0 -- Intelligent node provisioning with IRSA, SQS interruption queue, EventBridge rules for spot/health events
  • Cluster Autoscaler v9.35.0 -- ASG-based scaling with auto-discovery
  • Node Termination Handler v0.21.0 -- Graceful spot interruption draining, scheduled event draining, rebalance monitoring
  • Metrics Server v3.12.0 -- Resource metrics for HPA/VPA

Self-Healing Layer

  • Remediation Lambda -- Python 3.12 function triggered by EventBridge on EC2 state changes and health events
  • CloudWatch Alarms -- StatusCheckFailed alarms per node group
  • SNS Topic -- Node health alert notifications
  • Auto-Scaling Policies -- Scale-out (+2) and scale-in (-1) with 300s cooldown

Data Flow

  1. EKS cluster provisions with control plane logging to CloudWatch
  2. Managed node groups launch EC2 instances via launch templates into the VPC subnets
  3. EKS addons (CoreDNS, kube-proxy, VPC CNI, EBS CSI) install on cluster
  4. Karpenter watches for unschedulable pods and provisions optimal nodes via EC2 Fleet API
  5. SQS queue receives EC2 interruption/health events from EventBridge for Karpenter to handle gracefully
  6. Node Termination Handler detects spot interruptions and cordons/drains affected nodes
  7. CloudWatch alarms monitor EC2 StatusCheckFailed metrics and alert via SNS
  8. EventBridge captures EC2 state-change and health events, triggers remediation Lambda
  9. Lambda inspects the instance, terminates unhealthy nodes from ASG, and publishes to SNS
  10. ASG replaces terminated instances automatically, maintaining desired capacity

Industry Adaptation

Production Workloads

Use ON_DEMAND node groups for critical workloads with Karpenter for elastic compute. Enable auto_remediation for automatic recovery from node failures. Set min_size >= 2 for HA.

Cost-Optimized / Batch

Use SPOT capacity_type for fault-tolerant workloads. Enable Node Termination Handler for graceful spot handling. Use Karpenter for intelligent instance type selection.

Multi-Tenant Platforms

Define multiple node groups with labels and taints for tenant isolation. Use separate node groups for system workloads vs tenant workloads.

Regulated Industries

Enable full audit logging. Enforce IMDSv2. Use encrypted EBS volumes. Restrict public API access. Add network policies via Calico or Cilium post-deploy.

Production Readiness Checklist

  • Verify VPC has subnets in at least 2 AZs with proper tagging
  • Configure kubectl: aws eks update-kubeconfig --name <cluster>
  • Verify Karpenter pods running: kubectl get pods -n karpenter
  • Confirm metrics-server provides node metrics: kubectl top nodes
  • Test Node Termination Handler: kubectl get ds -n kube-system
  • Validate remediation Lambda in CloudWatch Logs
  • Subscribe alerting endpoints to the SNS node health topic
  • Set up Pod Disruption Budgets for critical workloads
  • Configure Karpenter NodePool and EC2NodeClass resources
  • Enable container insights for workload-level monitoring
  • Set up remote Terraform state backend (S3 + DynamoDB)
  • Tag all resources with Environment, Team, ManagedBy
